# Как работать над проектом

## Окружение

Для работы над проектом нужны **Node.js**, **npm** и доступная база данных **MongoDB**. Актуальные требования к версиям указаны в `package.json`: `node: ^18.0.0`, `npm: >=8`.

После перехода в корневую директорию проекта установите зависимости:

```bash
npm install
```

Команда загрузит все пакеты, необходимые для сборки, запуска REST API и работы вспомогательных утилит.

## Запуск проекта

Ниже приведён рекомендуемый порядок локального запуска проекта.

### 1. Установить зависимости

Если зависимости ещё не установлены, выполните:

```bash
npm install
```

### 2. Подготовить файл `.env`

Приложение читает настройки из файла `.env`. Создайте его в корне проекта и укажите основные переменные окружения:

```env
PORT=4000
SALT=123456
DB_HOST=127.0.0.1
DB_USER=admin
DB_PASSWORD=test
DB_PORT=27017
DB_NAME=buy-and-sell
UPLOAD_DIRECTORY=upload
JWT_SECRET=secret
HOST=localhost
STATIC_DIRECTORY_PATH=static
```

Назначение переменных:

- `PORT` - порт, на котором будет запущен REST API.
- `SALT` - соль для хеширования паролей.
- `DB_HOST`, `DB_USER`, `DB_PASSWORD`, `DB_PORT`, `DB_NAME` - параметры подключения к MongoDB.
- `UPLOAD_DIRECTORY` - директория для загружаемых файлов.
- `JWT_SECRET` - секрет для подписи JWT.
- `HOST` - хост приложения.
- `STATIC_DIRECTORY_PATH` - директория со статическими файлами.

### 3. Запустить MongoDB

Перед стартом приложения убедитесь, что сервер MongoDB запущен и доступен по параметрам из `.env`. Если база не будет доступна, REST API не сможет завершить инициализацию.

### 4. Собрать проект

```bash
npm run build
```

Команда очистит директорию `dist` и заново скомпилирует TypeScript-проект.

### 5. Запустить REST API

Для обычного запуска используйте:

```bash
npm start
```

Команда выполнит сборку и запустит сервер из `dist/main.rest.js`.

### 6. Запустить проект в режиме разработки

Во время разработки удобнее использовать:

```bash
npm run start:dev
```

Этот сценарий также пересобирает проект перед запуском, а затем поднимает сервер через `nodemon` и выводит форматированные логи через `pino-pretty`.

### 7. При необходимости поднять мок-сервер

Если нужен локальный источник тестовых данных, выполните:

```bash
npm run mock:server
```

После этого `json-server` поднимет мок-сервер на порту `3123`.

## Сценарии

В `package.json` предопределены следующие сценарии:

### `npm start`

Собирает проект и запускает REST API из `dist/main.rest.js`. Основной сценарий для локального запуска приложения.

### `npm run start:dev`

Собирает проект и запускает REST API в режиме разработки через `nodemon`. Подходит для разработки, когда важно автоматически перезапускать сервер и видеть читаемые логи.

### `npm run build`

Полная сборка проекта: сначала удаляет старую директорию `dist`, затем компилирует TypeScript-код заново.

### `npm run lint`

Проверяет код в директории `src` линтером **ESLint**. Используется после изменений и перед сдачей проекта, чтобы убедиться, что код соответствует правилам.

### `npm run compile`

Компилирует TypeScript в JavaScript по конфигурации из `tsconfig.json`. Удобно использовать, если очистка `dist` не требуется.

### `npm run clean`

Удаляет директорию `dist` с результатом предыдущей сборки. Обычно используется как подготовительный шаг перед новой компиляцией.

### `npm run ts -- <путь-к-файлу>`

Запускает TypeScript-файл напрямую через `ts-node` без предварительной сборки. Полезно для отладки отдельных модулей и запуска служебных скриптов во время разработки.

### `npm run cli -- <аргументы>`

Собирает проект и запускает CLI-приложение из `dist/main.cli.js`. Используется для работы с консольными командами проекта.

### `npm run mock:server`

Запускает `json-server` на порту `3123` с данными из `mocks/mock-server-data.json`. Подходит для локальной разработки и тестирования запросов без реального backend-источника данных.

## Структура проекта

### Директория `src`

Исходный код проекта: компоненты, модули и так далее. Структура директории `src` может быть произвольной.

### Файл `Readme.md`

Инструкции по работе с учебным репозиторием.

### Файл `Contributing.md`

Советы и инструкции по внесению изменений в учебный репозиторий.

### Остальное

Все остальные файлы в проекте являются служебными. Пожалуйста, не удаляйте и не изменяйте их самовольно. Только если того требует задание или наставник.
